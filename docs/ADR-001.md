# ADR-001: Document Intelligence Engine — Claude Vision API vs AWS Textract vs Google Document AI

**Status:** Accepted  
**Date:** April 25, 2026  
**Author:** Cesar Orellana — tradedocs-ai  
**Context:** Sprint 2 — Trade Document Intelligence extraction pipeline

---

## Context

Trade Document Intelligence requires extracting structured customs data from commercial invoices.
These documents vary widely in format, language, and quality — clean PDFs, scanned documents,
bilingual Spanish/English layouts, multi-page invoices, and documents from 10+ countries.

The extraction engine is the core technical component of the product. Choosing the right tool
determines accuracy, cost, complexity, and long-term maintainability.

Three options were evaluated:

1. **Claude Vision API** (Anthropic)
2. **AWS Textract** (Amazon Web Services)
3. **Google Document AI** (Google Cloud)

---

## Options Considered

### Option 1 — Claude Vision API (Anthropic)

**What it does:**  
Multimodal large language model that reads PDF/image content and responds to natural language
instructions. Can extract, interpret, reason, and format output in a single API call.

**Strengths:**
- Handles structured extraction AND semantic understanding in one step
- Natively bilingual — reads Spanish/English mixed documents without pre-processing
- Chain of thought reasoning — can self-verify extracted values before returning output
- Returns structured output directly (JSON, markdown tables, formatted lists)
- Zero-data-retention API option available — critical for handling sensitive trade documents
- No infrastructure setup — single API call, no pipelines, no post-processing
- Cost scales with usage — no minimum commitment

**Weaknesses:**
- Higher cost per document than raw OCR tools at scale
- Context window limits very long multi-page documents (mitigated with pdfplumber pre-extraction)
- Less deterministic than rule-based extraction — outputs can vary slightly between calls

**Cost estimate:** ~$0.003–$0.008 per invoice at current token pricing (claude-sonnet-4-6)

---

### Option 2 — AWS Textract

**What it does:**  
OCR service that detects and extracts text, forms, and tables from documents.
Returns raw text blocks and key-value pairs with bounding box coordinates.

**Strengths:**
- High accuracy on clean, structured documents
- Fast processing — optimized for throughput
- Integrates well with AWS ecosystem (S3, Lambda, DynamoDB)
- Predictable pricing at scale

**Weaknesses:**
- Returns raw text blocks — requires significant post-processing to extract semantic fields
- No native understanding of what "shipper" or "HS code" means — needs custom logic per document type
- Poor performance on bilingual documents — Spanish field labels require additional mapping
- No reasoning capability — cannot resolve ambiguous values or verify numerical consistency
- Requires building and maintaining a separate parsing layer on top of raw OCR output
- AWS account setup, IAM permissions, and infrastructure management overhead
- Data privacy: documents processed through AWS infrastructure — zero-data-retention not available by default

**Cost estimate:** ~$0.0015 per page for forms analysis + post-processing development cost

---

### Option 3 — Google Document AI

**What it does:**  
ML-based document processing platform with pre-trained processors for specific document types
(invoices, receipts, tax forms) and custom model training capability.

**Strengths:**
- Pre-trained invoice processor understands common invoice fields
- Higher accuracy than raw OCR for standard invoice formats
- Integrates with Google Cloud ecosystem
- Custom processors can be trained on specific document types

**Weaknesses:**
- Pre-trained invoice processor optimized for US/EU standard formats — poor performance on
  Latin American invoice layouts and Spanish-language field labels
- Custom processor training requires labeled dataset of 50–500 documents minimum
  before reaching production accuracy — not viable at current dataset size (10 documents)
- Complex setup: Google Cloud project, service accounts, API credentials, regional configuration
- No reasoning capability — cannot handle ambiguous values or cross-verify fields
- Higher operational complexity than a single API call
- Data residency concerns for sensitive trade documents

**Cost estimate:** ~$0.0065 per page for invoice processor + infrastructure setup time

---

## Decision

**Selected: Claude Vision API (Anthropic)**

---

## Rationale

### 1. Semantic understanding over raw OCR

AWS Textract and Google Document AI return raw text that must be mapped to customs fields
through custom post-processing logic. This logic must be maintained and updated every time a
new document format is encountered.

Claude Vision API understands the intent of the extraction request. When asked for the
"declared value and currency," it identifies the correct monetary value even when the document
labels it differently ("Total CIF," "Invoice Amount," "Valor Total") — without custom mapping code.

### 2. Bilingual document handling

The target market — Spanish-language trade documents across Latin America and Spain — produces
invoices with mixed Spanish/English field labels, Spanish date formats, and regional conventions.

In testing across 10 documents spanning 10 trade routes, Claude Vision correctly handled:
- Spanish dates: "18 de Abril de 2026" → automatically translated to "April 18, 2026"
- Bilingual descriptions: extracted goods descriptions in both languages simultaneously
- Regional currency formats: EUR and USD correctly identified and distinguished

AWS Textract and Google Document AI would require custom Spanish language mapping layers
to achieve equivalent results.

### 3. Chain of thought reasoning — self-verification

Prompt v2 implements a 4-step chain of thought before extraction:
1. Document overview — identifies trade route and document type
2. Section identification — locates party details, line items, weights, values
3. Ambiguity resolution — determines correct value when multiple candidates exist
4. Extraction — outputs fields with confidence

In testing, this approach led Claude to self-verify numerical fields:
"Net weight: 12,930.00 kg ✅ (verified: sum of all line items = 12,930 kg)"

No OCR-based tool can perform this reasoning. It requires an LLM.

### 4. Data privacy — zero-data-retention

Commercial invoices contain trade secrets: supplier relationships, pricing, product sourcing,
declared values. Clients handling this data require assurance it is not stored or used for
training.

Anthropic's API supports a zero-data-retention header (`anthropic-beta: zero-data-retention-1`)
that ensures documents are processed in memory only and not retained. This is enabled by default
in the tradedocs-ai extraction pipeline.

AWS Textract and Google Document AI do not offer equivalent zero-data-retention guarantees
without enterprise agreements.

### 5. Development velocity

Claude Vision API requires:
- One API call
- One prompt
- One response

AWS Textract or Google Document AI require:
- OCR extraction layer
- Text parsing and field mapping layer
- Confidence scoring layer
- Error handling for missed fields
- Separate logic per document format

For a solo founder working 15 hours per week, minimizing pipeline complexity is critical.
The simpler architecture ships faster, fails more predictably, and is easier to debug and iterate.

---

## Tradeoffs Accepted

| Tradeoff | Accepted because |
|----------|-----------------|
| Higher per-document cost vs raw OCR | Development time saved outweighs cost difference at current scale. Revisit at 1,000+ docs/day. |
| Less deterministic output vs rule-based | Chain of thought prompting and LLM-as-a-Judge evaluation (planned) mitigate this risk. |
| Dependency on single vendor | Anthropic SDK abstracts model — can switch to claude-opus or future models without pipeline changes. Mitigated by prompt versioning. |
| Context window limits on long documents | Mitigated by pdfplumber pre-extraction — text is extracted first, then sent to Claude. |

---

## Consequences

- All invoice extraction runs through the Anthropic Claude API (claude-sonnet-4-6)
- Zero-data-retention header is mandatory on all production API calls
- Prompt versioning is required — every prompt change must be logged with before/after accuracy
- LLM-as-a-Judge evaluation (Claude Haiku as critic) will be implemented in Sprint 4
  to validate semantic accuracy beyond string matching
- Cost per document must be tracked per user to ensure pricing covers API costs
- If per-document volume exceeds 1,000/day, revisit AWS Textract as pre-processing layer
  to reduce token consumption before Claude reasoning step
